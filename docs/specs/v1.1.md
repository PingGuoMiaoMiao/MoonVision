# MoonVision v1.1 Spec

## Status

- Phase: draft
- Base branch: `v1.0`
- Goal: add a small set of high-value, OpenCV-inspired image processing features without expanding into video, GUI, or ML scope

## Why v1.1

`v1.0` already covers the minimum algorithm layer:

- image containers
- basic pixel ops
- blur and sharpen filters
- Sobel-based edge extraction
- binary morphology
- connected components
- PNG/SVG/HTML export

The next iteration should improve practical utility and comparability with classic `imgproc` workflows from OpenCV, while staying small enough to verify and demo clearly.

`v1.1` is not only a feature iteration. It is also the first optimization-oriented iteration:

- improve robustness on uneven illumination
- reduce noise sensitivity before binary segmentation
- improve demo output quality without changing the overall library scope
- keep all improvements measurable against the `v1.0` baseline

## Scope

### In Scope

1. Automatic thresholding
- `otsu_threshold`
- keep existing fixed threshold API unchanged

2. Local thresholding
- `adaptive_threshold_mean`
- optional `adaptive_threshold_gaussian` only if the mean variant lands cleanly without destabilizing schedule

3. Non-linear denoising
- `median_blur`

4. Basic geometric transforms
- `resize_nearest`
- `flip_horizontal`
- `flip_vertical`
- `rotate90_cw`
- `rotate90_ccw`

5. Demo improvements
- object counting demo gets an automatic-threshold path
- document enhancement demo gets an adaptive-threshold path

6. Baseline comparison support
- keep the current `v1.0` sample assets stable for comparison
- record before/after result summaries in docs
- use the same demo entrypoints so `v1.0` and `v1.1` remain directly comparable

### Out of Scope

- Canny
- contour extraction
- perspective transform
- histogram equalization
- affine transform
- color-space expansion beyond current RGB/gray flow
- video, GUI, OpenCV bindings, DNN, ML

## API Direction

Follow the existing MoonVision style:

- pure functions first
- same input/output image types as `v1.0`
- explicit argument names for tunable parameters
- no silent fallback to panic-based error handling

Proposed public APIs:

```moonbit
pub fn otsu_threshold(image : GrayImage) -> GrayImage!MoonVisionError

pub fn adaptive_threshold_mean(
  image : GrayImage,
  block_size : Int,
  c : Int
) -> GrayImage!MoonVisionError

pub fn median_blur(
  image : GrayImage,
  radius : Int
) -> GrayImage!MoonVisionError

pub fn resize_nearest(
  image : GrayImage,
  new_width : Int,
  new_height : Int
) -> GrayImage!MoonVisionError

pub fn flip_horizontal(image : GrayImage) -> GrayImage!MoonVisionError
pub fn flip_vertical(image : GrayImage) -> GrayImage!MoonVisionError
pub fn rotate90_cw(image : GrayImage) -> GrayImage!MoonVisionError
pub fn rotate90_ccw(image : GrayImage) -> GrayImage!MoonVisionError
```

`RgbImage` geometric transforms may be added in the same iteration if implementation stays straightforward. If not, gray support is the required minimum and RGB support moves to a follow-up spec.

## Design Constraints

- Keep all new algorithms MoonBit-native
- Reuse current image container layout
- Do not redesign the error model in `v1.1`
- Preserve current demo output contract under `examples/output/`
- Avoid speculative abstraction for future kernels or transform backends
- Prefer correctness and stable output over premature micro-optimization
- Any optimization claim must be backed by the same asset set used in `v1.0`

## Optimization Targets

The optimization goal of `v1.1` is quality-first, not raw throughput-first.

Primary targets:

1. Thresholding robustness
- `adaptive_threshold_mean` should outperform fixed thresholding on uneven illumination inputs
- `otsu_threshold` should reduce manual parameter dependence in the object counting flow

2. Noise handling
- `median_blur` should remove isolated salt-and-pepper noise with less edge damage than a Gaussian blur at similar neighborhood size

3. Demo stability
- object counting output should require fewer asset-specific magic constants
- document enhancement output should remain readable under mild shadows and local contrast variation

Secondary targets:

- keep runtime acceptable on small and medium demo images
- avoid introducing additional dependency layers or backend-specific code paths

## Risks

1. Adaptive threshold parameter sensitivity
- Different images may need different `block_size` and `c`
- Mitigation: lock one default demo configuration and test a few representative assets

2. Median blur cost
- A naive neighborhood sort is slower than linear filters
- Mitigation: keep radius support small in `v1.1` and validate correctness before optimization

3. Resize semantics
- Introducing interpolation choices too early will expand API surface
- Mitigation: `v1.1` only guarantees nearest-neighbor resize

4. Scope creep
- Adding Canny or contour extraction in the same pass would broaden verification too much
- Mitigation: keep them explicitly out of scope for this spec

5. Optimization ambiguity
- It is easy to claim an image “looks better” without stable evidence
- Mitigation: compare `v1.0` and `v1.1` on the same assets and record a short result table in documentation

## Validation

Required checks for implementation approval:

- `moon check`
- `moon test`
- `moon run src/demo/object_counting`
- `moon run src/demo/document_enhancement`

Comparison checks:

- keep at least one `v1.0`-style fixed-threshold output for side-by-side inspection
- compare `v1.0` and `v1.1` object counts on the same demo asset
- compare `v1.0` and `v1.1` document outputs on the same demo asset

Required test additions:

- Otsu threshold returns stable split on fixed bimodal samples
- Adaptive threshold handles uneven illumination samples
- Median blur removes isolated salt-and-pepper noise better than raw threshold input
- Resize and rotate preserve expected output dimensions and pixel mapping
- Flip operations are exact mirror transforms

## Acceptance Criteria

`v1.1` is complete when all of the following are true:

1. The new APIs compile and are covered by unit tests
2. `moon test` passes on the repository default target
3. Object counting demo can run with automatic thresholding
4. Document enhancement demo can run with adaptive thresholding
5. README documents the new APIs and demo behavior changes
6. Repository documentation includes a short `v1.0` vs `v1.1` comparison summary for the upgraded demos

## Commit Strategy

Target 6 to 9 additional meaningful commits for `v1.1`, for example:

1. refine spec and comparison criteria
2. add Otsu threshold with focused tests
3. add adaptive threshold with uneven-lighting tests
4. add median blur with salt-and-pepper tests
5. add geometric transforms for `GrayImage`
6. extend geometric transforms to `RgbImage` if it stays low-risk
7. upgrade object counting demo to use automatic thresholding
8. upgrade document enhancement demo to use adaptive thresholding
9. update README with `v1.0` vs `v1.1` comparison and run final verification

Recommended execution rule:

- each commit should add one coherent capability or one coherent verification/documentation step
- avoid mixing algorithm work and demo rewiring in the same commit unless one depends tightly on the other
- keep tests in the same commit as the feature they validate
- only split further when the intermediate state is still reviewable and testable

Each commit must produce a coherent, testable unit of progress.
